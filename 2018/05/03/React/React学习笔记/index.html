React学习笔记

发表于 2018-05-03 | 更新于 2018-05-03 | 分类于 前端 ， React

JSX

JavaScript的语法扩展

在JS中编写HTML一样的代码，JSX中使用的‘元素’不局限与HTML中的元素，可以是任何一个React组件。React判断一个元素是HTML元素还是React组件的原则是看第一个字母是否大写。

JSX的onClick事件处理方式和HTML的onclick不同。

  • HTML的onclick处理方式

    onclick添加的事件处理函数是在全局环境下执行的，会污染全局环境，容易产生意料不到的后果

    过多的onclick事件，可能会影响网页的性能

    需要动态删除使用onclick的DOM元素时，需要把对应的时间处理器注销。如果忘记注销，会造成内存泄漏。

  • JSX的onClick处理方式

    不产生直接使用onclick的HTML，而是使用事件委托的方式处理点击时间。

    在最顶层的DOM节点上，添加一个事件处理函数。点击的事件都被该处理函数捕获，然后根据具体组件分配给特定函数。

    React控制组件的生命周期，在unmount的时候能清除相关的所有事件函数，不会发生内存泄漏。

React

理念

UI=render(data)

这个函数是一个纯函数，即没有任何副作用，输出完全依赖于输入的函数。

DOM

DOM操作会引起浏览器对页面进行重新布局，重新绘制

Virtual DOM

React首先构建Virtual DOM。Virtual DOM不会触及浏览器的部分，只存在与Javascript空间的树形结构，每次自上而下渲染React组件时，会对比这次产生的和上次渲染的Virtual DOM，对比就会发现差别，然后修改真正的DOM树时就只触及差别中的部分就行。

构造函数

如果一个组件需要构造函数，第一行必须调用父类的构造函数，否则类实例的所有成员函数就无法通过this.props访问到父组件传递过来的props值。

成员函数与实例的绑定

ES6方法创造的React组件类不会自动绑定this到当前实例对象，所以需要在构造函数中，做显示的绑定

propTyps检查

声明自己接口规范，包含以下方面：

  • 支持的prop
  • 每个prop的格式（类型）

这是一种限制，在运行时和静态代码检查时，都可以根据prop Types判断。

PropTypes已从最新版的React库中移除，单独作为一个库，所以在使用的时候需要再单独引入

state和prop的局限

1.当数据重复时，由于存了多分，如果出现数据不一致，将无法判断以哪个为准

2.包含多层组件结构时，顶层组件想传递数据给最底层子组件时，用prop的方式，就只能通过中间的组件层中转。但也许中间层并不需要这些数据，但依然需要支持这个prop。

Flux

四个组成部分：

  • Dispatcher，处理动作分发，维持Store之间的依赖关系
  • Store，负责存储数据和处理数据相关逻辑
  • Action，驱动Dispatcher的JavaScript对象
  • View，视图部分，负责显示用户界面

Dispatcher

一个应用只维持唯一的Dispatcher对象，从Flux中import

Action

  • action.js
  • actionType.js

action.js，定义产生并派发action的对象函数

actionType.js，定义action类型

Store

存储应用状态，同时接受Dispatcher派发的动作，根据动作来决定是否更新应用状态。可以有多个Store对象。Store对象要注册到Dispatcher对象上。

优点

数据不再由View修改，而是在Store中。

缺点

1.Store之间的依赖关系

一个action会派发给所有回调函数，而且Store的装载顺序无法预期，从而Dispatcher调用回调函数的顺序也就无法预期。所以需要使用Dispatcher的waitFor函数。

2.难以进行服务器渲染

在服务器端渲染，输出不是一个DOM树，而是一个字符串（全是HTML的字符串）

Dispatcher和Store都是全局唯一对象，不同请求的状态就全乱套了。

3.Store混乱了逻辑和状态

Redux

基本原则

  • 唯一数据源

    应用的状态数据应该只存储在唯一的一个Store上

  • 保持状态只读

  • 数据改变只能通过纯函数完成

    reduce(state, action)

    根据state和action生成一个新的state对象并返回，不去修改入参
